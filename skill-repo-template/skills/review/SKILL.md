# 代码审查

## 概述

提供专业的代码审查能力，通过五大审查维度系统性检查代码变更，自动读取项目配置进行版本感知的审查。

## 前置步骤（必须执行）

1. **读取项目上下文**：使用 Read 工具读取 `.claude/project-context.md`
   - 确认技术栈、版本信息、开发规范
   - 对版本兼容性检查至关重要
2. **读取 package.json**：确认当前项目的依赖版本
3. **理解项目规范**：根据配置文件了解代码规范和特殊约定

## 审查流程

1. **获取变更内容**：`git diff` / `git diff --cached` / `git show HEAD`
2. **识别变更范围**：修改了哪些文件？涉及哪些模块？变更类型？
3. **执行五维度审查**
4. **生成结构化审查报告**

## 五大审查维度

### 一、样式覆盖检查
CSS/Less 全局污染、UI 库样式覆盖、CSS Module 使用、样式文件组织

### 二、第三方库版本检查
React 版本兼容性、UI 库 API 差异、路由库版本差异、状态管理差异

### 三、影响范围分析
文件依赖、API 变更、全局影响、跨版本影响、数据流影响

### 四、代码质量检查
React 最佳实践、性能问题、安全性、错误处理、代码规范

### 五、测试相关
功能完整性、边界情况、测试建议

## 输出契约

审查结果先给 Findings，再给总结：
1. **Findings**（按严重度从高到低），每条含：级别、文件路径、行号、问题、影响、建议
2. **Open Questions**（如有）
3. **总结**（最多 5 行）

未发现阻断问题时，明确写出"未发现阻断问题"，并给出残余风险和测试缺口。

## 核心原则

- **聚焦关键问题**：不吹毛求疵，关注真正重要的问题
- **建设性反馈**：不仅指出问题，还要给出解决方案
- **量化评估**：使用明确的风险等级和评分
- **高效审查**：优先检查高风险区域

## 审查重点因变更类型而异

| 变更类型 | 审查重点 |
|---------|---------|
| 新功能 | 样式覆盖、影响范围、性能 |
| Bug 修复 | 影响范围、是否引入新问题 |
| 重构 | 兼容性、影响范围 |
| 样式修改 | 样式覆盖、全局污染 |

## 工作流衔接

| 上游 | 下游 |
|------|------|
| `/code-by-plan` 编码完成 | → 提交代码、创建 PR |
| 手动代码修改 | → `/bug-fix` 修复发现的 bug |
