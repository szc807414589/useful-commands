# Bug 修复

## 概述

本 skill 提供从问题定位到修复验证的完整 Bug 修复流程。通过系统化的六阶段方法确保找到根本原因、全面评估影响、治本修复并充分验证。

## 前置步骤（必须执行）

1. **读取项目上下文**：使用 Read 工具读取 `.claude/project-context.md`，确认技术栈、版本信息
2. **读取 package.json**：确认当前项目的依赖版本
3. **理解用户描述**：Bug 表现、出现场景、预期行为、实际行为

## 执行模式

### 标准模式（默认）
适用于：根因不清晰、跨模块、存在回归风险的 bug。执行完整六阶段流程。

### 轻量模式
适用于：复现路径明确、根因局部且修复面很小的 bug。可缩短方案比较和文档描述，但不能跳过验证。

## 六阶段流程

### 阶段 1：问题定位

**目标**：找到根本原因，不停留在表面现象

1. 理解问题表象（症状、复现步骤、报错信息）
2. 搜索相关代码（组件、函数、数据流）
3. 深入追踪完整调用链（触发点 -> 中间环节 -> 问题点）
4. 寻找参考实现（项目中类似问题的处理方式）

**关键**：使用 `rg` 搜索函数调用关系，记录具体证据（文件路径 + 行号）。

### 阶段 2：影响分析

**目标**：全面评估修复的影响范围

1. 直接影响：使用 `rg` 搜索所有引用点
2. 间接影响：数据流、API、样式、路由
3. 风险评估：复杂度、影响范围、引入新 bug 的可能性

### 阶段 3：方案设计

**目标**：设计治本而非治标的方案

1. 生成根因分析（表面现象、触发条件、完整调用链、根本原因、证据）
2. 方案对比（如有多种修复方式）：

| 方案 | 类型 | 优点 | 缺点 | 是否采用 |
|------|------|------|------|---------|
| 方案 A | 治标 | 快速 | 不解决根本问题 | 不推荐 |
| 方案 B | 治本 | 从根源解决 | 改动稍大 | 推荐 |

3. 详细修复方案（思路、文件、修改点、参考实现）
4. 向用户确认方案

### 阶段 4：执行修复

**目标**：严格按方案执行

1. 创建任务清单跟踪修复步骤
2. 逐步修复（标记进度 → 读取 → 定位 → 修改 → 完成）
3. 处理边界情况（空值、异常、输入验证、竞态条件）
4. 代码质量检查（规范、重复、性能、安全）

### 阶段 5：验证测试

**目标**：确保修复有效且不引入新问题

1. 构建验证（编译错误、类型错误）
2. 功能验证（正常流程、边界情况）
3. 回归测试（受影响的相关功能）

### 阶段 6：总结交付

**目标**：完整记录修复过程

使用修复报告模板生成报告，包含：问题概述、根因分析、修复方案、验证结果、影响评估、建议测试点。

## 核心原则

1. **探索 > 假设**：通过代码探索找到真相，不基于假设下结论
2. **治本 > 治标**：问"为什么会这样"，而非"怎么绕过去"
3. **参考 > 创造**：优先使用项目中已有的模式和工具
4. **全面 > 片面**：分析影响范围，搜索所有相关代码
5. **验证 > 假定**：修复后必须验证，不假定修复一定有效

## 特殊场景

| 场景 | 特点 | 处理要点 |
|------|------|---------|
| 紧急 Bug | 线上问题，需快速修复 | 先治标后治本，记录技术债务 |
| 复杂 Bug | 多个原因交织 | 拆解子问题，逐个分析，综合修复 |
| 历史遗留 | 代码年代久远 | 先理解设计意图，优先向后兼容 |
| 间歇性 Bug | 难以复现 | 添加日志，分析触发条件，模拟复现 |

## 精简检查清单

- [ ] 找到根本原因（有证据）
- [ ] 评估了影响范围（直接 + 间接）
- [ ] 方案是治本的（或紧急场景有后续治本计划）
- [ ] 与用户确认了方案
- [ ] 修复后通过构建
- [ ] 测试了正常流程和边界情况
- [ ] 生成了修复报告

## 下游衔接

- 修复完成后，建议使用 `/review` 审查修复代码
- 如果 bug 涉及架构问题，建议使用 `/design` 重新设计
